# 패치노트

이 디렉토리는 프로젝트의 데일리 패치노트를 기록하는 공간입니다.

## 디렉토리 구조

패치노트는 주차별로 폴더에 정리됩니다.

```
patch/
  week-2026-01-05/          # 주차별 폴더 (월요일 날짜)
    README.md               # 주차별 요약
    day2-01-*.md            # 일별 패치노트
    day3-01-*.md
    ...
  week-2026-01-12/          # 다음 주차
    ...
```

### 주차별 폴더 명명 규칙

- 형식: `week-YYYY-MM-DD` (해당 주의 월요일 날짜)
- 예: `week-2026-01-05` (2026년 1월 5일이 월요일인 주)

## 작성 방법

각 날짜별로 패치노트를 작성합니다. 버전이 추가된 날에는 파일명에 버전을 함께 표기합니다.

### 파일 명명 규칙

주차별 폴더 내부의 패치노트 파일은 다음과 같은 형식을 사용합니다:

- 기본 형식: `dayN-번호-요약.md` (예: `day2-01-프로젝트초기설정.md`, `day3-02-ApiResponseTemplate구현.md`)
  - `dayN`: 주차 내 일차 번호 (day1, day2, day3, ...)
  - `번호`: 같은 날 여러 패치노트를 작성할 경우 증가 (01, 02, ...)
  - `요약`: 간단한 설명
- 버전 추가 형식: `dayN-번호-Release-v0.0.1.md` (예: `day2-01-Release-v0.0.1.md`)

**일차 번호:**
- 주차의 월요일부터 시작하여 day1, day2, day3, ... 순서로 증가
- 예: 월요일=day1, 화요일=day2, 수요일=day3, ...

같은 날 여러 패치노트를 작성할 경우 번호를 증가시켜 구분합니다. 파일명에 간단한 설명을 포함하여 내용을 쉽게 파악할 수 있도록 합니다. 버전이 릴리즈된 날에만 버전을 파일명 뒤에 추가합니다.

### 주차별 요약 파일

각 주차 폴더에는 `README.md` 파일을 생성하여 해당 주차의 주요 작업 내용을 요약합니다.

### 패치노트 템플릿

각 패치노트는 아래 구조를 따릅니다. 프론트매터로 메타정보를 두고, 요약 → 문제/목표 → 설계·결과 → 구현 상세 순으로 작성합니다.

```markdown
---
title: (작업 제목)
tags: [개인 프로젝트, (관련 키워드)]
date: YYYY-MM-DD
---

## 프로젝트 요약

- **한 줄 요약**: (이번 작업을 한 문장으로)
- **키워드**: `(키워드1)`, `(키워드2)`

## 문제(AS-IS)

- (해결하려 했던 문제나 개선 포인트)

## 목표(TO-BE)

- (이번 작업으로 달성하려 한 목표)

## 설계/선택(Key decisions)

(주요 설계 선택·후보 비교·선택 이유가 있으면 기록. 없으면 생략 또는 "구현 후 채울 예정")

## 결과(Impact)

(변경/개선 결과 요약. 구현 후 채우기)

---

## 구현 상세

**목차**
1. (작업 1 제목)
2. (작업 2 제목)

---

### 1) (작업 1 제목)

**배경**: (왜 이 작업을 했는지)

(해결방안 후보가 있으면 나열 후 **선택**과 **이유**·**트레이드오프** 정리. 없으면 **작업 내용**·**변경 클래스**·**작업 결과** 등으로 구체적 변경 사항 기록)

---

### 2) (작업 2 제목)

**목적**: (해당 항목의 목적)

(동일하게 배경·후보·선택·플로우 등 필요한 만큼 작성)
```

- **프론트매터**: `title`, `tags`, `date`로 문서 메타정보를 둡니다.
- **프로젝트 요약**: 한 줄 요약과 키워드로 검색·파악이 쉽게 합니다.
- **문제(AS-IS) / 목표(TO-BE)**: 인과관계를 명확히 합니다.
- **설계/선택**: 후보 비교·선택 이유·트레이드오프를 남기면 이후 검토에 유리합니다.
- **구현 상세**: 항목별로 **배경** → (후보·선택)·**작업 결과** 순으로, 필요 시 목차를 두고 `### N)` 형태로 나눕니다.
